# 项目追踪

最后更新: 2026-04-05

## 当前实际情况

- 项目当前主入口是 [main.py](/D:/work/AI/yolo-study-001/main.py)，视觉主循环在 [vision.py](/D:/work/AI/yolo-study-001/vision.py)。
- 控制器工厂在 [controller.py](/D:/work/AI/yolo-study-001/controller.py)，支持 `gamepad`、`kbm_to_gamepad`、`mouse` 三种模式。
- 主流程已经去掉过渡实现，只保留一套正式代码路径。
- 视觉侧当前默认使用 `yolo26n-pose.engine`，失败时回退到 `yolo26n-pose.pt`。
- 模型、ONNX、engine 文件目前直接放在仓库根目录，还没有做统一配置管理。
- 项目目前没有 `README`、没有自动化测试、没有独立的配置文件。
- 仓库里存在一些独立脚本和运行遗留文件，比如 `guaji.py`、`start.bat`、`__pycache__`，暂时没有纳入正式运行主线。

## 已完成改动

- 入口整理:
  - `main.py` 现在支持 `--controller-mode`、`--perf-log`、`--crop-size`、`--target-fps`。
- 运行配置整理:
  - `vision.py` 支持从环境变量读取运行参数，避免反复改源码。
- 热路径优化:
  - 截图线程改成“最新帧缓冲”，替代旧的队列清空逻辑。
  - YOLO 检测结果统一解析后复用，避免多个模块重复取 `cpu().numpy()`。
  - 颜色判断改成整帧一次 HSV/mask 预处理，减少每个候选框重复转换颜色空间。
- 稳定性改进:
  - 控制器初始化失败会明确报错，不再返回半初始化对象。
  - 主流程退出时会 `reset + stop + join` 控制器线程，清理更完整。
- 可观测性:
  - 支持通过 `VISION_PERF_LOG=1` 或 `--perf-log` 打开周期性性能日志。

## 已知限制

- 目前还是偏“脚本型项目”，缺少面向长期维护的文档、配置和测试。
- 模型文件仍散落在仓库根目录，替换模型和切换版本不够直观。
- 运行状态主要靠控制台输出，用户很难快速判断当前是否成功启用、是否在辅助、是否退回了 `.pt`。
- 参数调优仍然需要手动试，不同游戏/灵敏度下迁移成本较高。

## 用户视角需求清单

### P0

- 配置文件化
  - 目标: 用户不改源码也能切模型、切控制器、改 crop、改阈值、改灵敏度。
  - 建议形式: `toml` 或 `json`。
- 启动自检
  - 目标: 启动时明确提示当前控制器、模型加载结果、是否使用 engine、是否检测到手柄、当前关键参数。
- 真实场景验证
  - 目标: 在真实使用链路里确认延迟、稳定性和锁定体验，不只停留在静态代码优化。

### P1

- 状态提示层
  - 目标: 用户能看到“是否启用辅助、是否在瞄准、是否锁到目标、当前性能”。
  - 形式可选: 小窗、overlay、控制台精简状态行。
- 动态 ROI / 自适应搜索范围
  - 目标: 丢目标时扩大搜索，锁目标时缩小推理区域，提升体感延迟和稳定性。
- 更系统的性能采样
  - 目标: 记录 capture / infer / post / controller 各阶段耗时，便于定位瓶颈。

### P2

- 标定/校准流程
  - 目标: 根据不同游戏和玩家习惯快速调出合适的 smooth、deadzone、AI force。
- 热切换能力
  - 目标: 运行时切换模式、开关日志、快速启停，不必重启整个程序。
- 目录整理
  - 目标: 把模型、脚本、文档、临时文件拆分到更清晰的目录结构。

## 建议的下一步顺序

1. 配置文件化。
2. 启动自检。
3. 真实场景验证并记录性能。
4. 状态提示层。
5. 动态 ROI。
